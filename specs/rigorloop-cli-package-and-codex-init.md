# RigorLoop CLI Package and Codex Init

## Status

approved

## Related proposal

- [RigorLoop Scaffolding CLI and Machine-Readable Workflow](../docs/proposals/2026-05-15-rigorloop-scaffolding-cli-and-machine-readable-workflow.md)

## Goal and context

This spec defines only the first implementable CLI slice for RigorLoop: one npm package candidate, one `rigorloop` binary, command discovery, version reporting, and `init --adapter codex` with safe project scaffolding and verified Codex adapter release-archive installation.

The broader proposal remains the product direction, but this spec intentionally does not define `new-change`, `status`, `validate`, durable lockfile writes, workflow YAML canonicality, generated workflow docs, npm publication hardening, or non-Codex adapters.

The behavior here is compatibility-sensitive because it creates public command syntax, machine-readable output, mutation safety rules, adapter archive verification rules, and the first `rigorloop.yaml` shape.

## Glossary

- `CLI package`: the npm package candidate named `@xiongxianfei/rigorloop`.
- `rigorloop binary`: the executable exposed by the CLI package as `rigorloop`.
- `Codex adapter`: the generated public Codex adapter package installed under `.agents/skills`.
- `release archive`: a Codex adapter ZIP asset attached to an official GitHub release for `xiongxianfei/rigorloop`.
- `release metadata`: the structured metadata file that identifies official adapter archives, hashes, size, install root, tree hash, and validation result.
- `bundled adapter metadata`: official release metadata shipped inside the installed `@xiongxianfei/rigorloop` package for the package's compatible adapter release.
- `planned write`: an intended file or directory mutation reported before it is applied.
- `planned lockfile content`: lockfile-shaped JSON output that previews what a future lockfile could contain without writing `rigorloop.lock`.
- `generated output`: adapter files extracted from a verified release archive; generated output is not authored source.
- `user file`: an existing file that is not known to be generated by this command invocation.
- `official release source`: GitHub Releases for `https://github.com/xiongxianfei/rigorloop`.

## Examples first

### Example E1: help discovers the first command surface

Given the CLI package is installed or executed through a package runner
When the user runs `rigorloop --help`
Then stdout shows human-readable usage for `rigorloop`, `rigorloop version`, and `rigorloop init --adapter codex`
And the command exits with code `0`.

### Example E2: version reports package identity

Given the installed package version is `0.1.3`
When the user runs `rigorloop version`
Then stdout includes `@xiongxianfei/rigorloop` and `0.1.3`
And the command exits with code `0`.

### Example E3: dry-run JSON plans Codex init without writing files

Given a project directory does not contain `rigorloop.yaml` or `.agents/skills`
When the user runs `rigorloop init --adapter codex --dry-run --json`
Then stdout contains one JSON object using schema version `1`
And the JSON reports planned creation of `rigorloop.yaml` and the Codex adapter install root
And the JSON includes `planned_lockfile`
And no files are written.

### Example E4: pinned package version installs matching Codex archive

Given the CLI is executed as `npx @xiongxianfei/rigorloop@0.1.3 init --adapter codex`
When bundled official adapter metadata for `v0.1.3` identifies a Codex adapter archive with matching checksum, size, install root, tree hash, and validation result `pass`
Then the command creates or updates only allowed scaffold and generated adapter files
And it does not write `rigorloop.lock`
And it exits with status `success` or `warning`.

### Example E5: local archive installs without network download

Given the user has `./rigorloop-adapter-codex-v0.1.3.zip`
When the user runs `rigorloop init --adapter codex --from-archive ./rigorloop-adapter-codex-v0.1.3.zip`
Then the command verifies the archive against bundled adapter metadata for `v0.1.3` before extraction
And extraction is confined to `.agents/skills`.

### Example E6: overwrite conflict is refused

Given `.agents/skills/proposal/SKILL.md` already exists and is not part of the current verified generated-output plan
When the user runs `rigorloop init --adapter codex`
Then the command refuses to overwrite the file
And exits with code `5`
And reports a blocker or error that names the conflicting path.

### Example E7: lockfile is never durably written in this slice

Given `init --adapter codex` completes
When the command reports expected adapter archive and tree hashes
Then the report may include `planned_lockfile`
But `rigorloop.lock` is not created or modified.

## Requirements

### Package and command surface

R1. The first CLI package candidate MUST be named `@xiongxianfei/rigorloop`.

R2. The package MUST expose exactly one public binary for this slice: `rigorloop`.

R3. The package `bin` contract MUST map `rigorloop` to the built CLI entrypoint.

R4. The in-scope command surface MUST be limited to:

```text
rigorloop --help
rigorloop version
rigorloop init --adapter codex
rigorloop init --adapter codex --dry-run --json
```

R5. `rigorloop --help` MUST exit `0` and describe only implemented or explicitly out-of-scope command categories without implying that `new-change`, `status`, or `validate` are available.

R6. `rigorloop version` MUST exit `0` and print the package name and package version.

R7. Unknown commands MUST exit `4` and produce an actionable usage error.

R8. Unsupported adapters MUST be rejected with status `blocked` and exit code `2`.

### JSON and exit contract

R9. When `--json` is supplied, stdout MUST contain JSON only.

R10. JSON output MUST use this top-level envelope:

```json
{
  "schema_version": 1,
  "command": "init",
  "package": {
    "name": "@xiongxianfei/rigorloop",
    "version": "0.1.3"
  },
  "cwd": "/path/to/project",
  "status": "success",
  "summary": "",
  "actions": [],
  "artifacts": [],
  "blockers": [],
  "warnings": [],
  "errors": [],
  "diagnostics": {}
}
```

R11. JSON `status` MUST be one of `success`, `warning`, `blocked`, or `error`.

R12. Exit codes MUST use this mapping:

| Exit code | Meaning |
|---|---|
| `0` | success or warning |
| `2` | blocked |
| `3` | validation failed |
| `4` | invalid usage or invalid config |
| `5` | mutation conflict or overwrite refused |
| `1` | internal or unexpected error |

R13. `actions` entries MUST use stable fields `type`, `path`, `status`, and `reason`.

R14. `artifacts` entries MUST use stable fields `path`, `kind`, and `status`.

R15. `blockers`, `warnings`, and `errors` entries MUST use stable fields `code`, `message`, `path` when path-specific, and `next_action` when a useful next action exists.

R16. In human mode, stdout MUST contain human-readable command output and MUST NOT print JSON fragments as routine output.

R17. Human diagnostics and error details MAY use stderr, but stderr MUST NOT be mixed into JSON stdout.

R18. `--debug` MAY add diagnostic detail under `diagnostics` in JSON mode and MAY print additional stderr diagnostics in human mode, but it MUST NOT remove or rename stable top-level JSON fields.

R19. `--quiet` MUST suppress non-essential human output and MUST NOT change command behavior, exit codes, or JSON shape.

R20. `--no-color` and the `NO_COLOR` environment variable MUST suppress ANSI color sequences in human output.

### Init inputs

R21. `rigorloop init` MUST require `--adapter codex` in this slice.

R22. `rigorloop init --adapter codex` MUST default to the current working directory as the target project root.

R23. `rigorloop init --adapter codex` MAY support an explicit target directory only if the command applies the same mutation safety and path traversal rules to that target.

R24. `--from-archive <path>` MUST select local archive installation mode.

R25. Network release-archive download MUST be used only when no `--from-archive` path is supplied.

R26. A pinned package version such as `0.1.3` MUST resolve to the matching adapter release tag `v0.1.3` by default.

R27. A prerelease package version MUST resolve to the matching prerelease release tag by default.

R28. A package version executed as `latest` MUST resolve to the concrete installed CLI package version before adapter compatibility is evaluated.

R29. `latest` with `--from-archive` MUST block if the concrete installed CLI package version expects a different adapter release and no explicit compatibility rule exists.

R29a. Version or release incompatibility MUST produce status `blocked`, exit code `2`, and blocker code `release-version-incompatible`.

### Basic `rigorloop.yaml`

R30. Actual non-dry-run init MUST create `rigorloop.yaml` when it does not exist and no blocker prevents initialization.

R31. `--dry-run` MUST report the planned `rigorloop.yaml` content without creating or modifying the file.

R32. `rigorloop.yaml` generated by this slice MUST use `schema_version: 1`.

R33. `rigorloop.yaml` generated by this slice MUST record the selected adapter `codex` and install root `.agents/skills`.

R34. `rigorloop.yaml` generated by this slice MUST record that the adapter source is a release archive or local archive.

R35. `rigorloop.yaml` generated by this slice MUST NOT claim branch readiness, PR readiness, workflow acceptance, validation success, or lockfile authority.

R36. If validation commands are generated or reserved in `rigorloop.yaml`, their names MUST be limited to `selected`, `ci`, `change_metadata`, `review_artifacts`, `artifact_lifecycle`, and `skills`.

R37. Generated validation command entries MUST NOT point to nonexistent project-local scripts unless they are explicitly marked inactive or example-only.

R37a. `rigorloop.yaml` generated by this slice MUST use this minimum shape for a release-archive install:

```yaml
schema_version: 1
rigorloop:
  package: "@xiongxianfei/rigorloop"
  package_version: "0.1.3"
adapters:
  - name: codex
    install_root: ".agents/skills"
    source:
      type: release-archive
      release: "v0.1.3"
```

R37b. `rigorloop.yaml` generated by this slice MUST use `source.type: local-archive` when `--from-archive <path>` is used.

R37c. First-slice generated `rigorloop.yaml` MUST NOT include `validation.commands` by default.

### Mutation safety

R38. `init` MUST default to non-destructive behavior.

R39. `--dry-run` MUST perform no file writes, directory creation, archive extraction, or deletion.

R40. Before mutating, `init` MUST compute a write plan that identifies every file and directory it intends to create, update, skip, or block.

R41. `init` MUST refuse to overwrite existing user files by default.

R42. `init` MUST NOT delete user files in this slice.

R43. `--force` MUST NOT allow replacement of arbitrary user files in this slice.

R44. Any allowed replacement under `--force` MUST be limited to documented generated outputs that the command can identify as generated by the same adapter install plan.

R45. Overwrite refusal MUST exit with code `5`.

R46. Archive extraction MUST reject absolute paths, parent-directory traversal, empty paths, drive-letter paths, and symlink entries.

R47. Extracted paths MUST remain under the expected install root `.agents/skills`.

R48. Partial installation failure MUST leave a JSON or human-visible error that identifies whether scaffold files, adapter files, or both may have been written.

### Codex adapter release metadata and archive verification

R49. Networked adapter installation MUST use bundled official adapter metadata for the installed CLI package version, then download the official adapter archive URL named by that metadata.

R50. Local archive installation MUST verify the archive against bundled adapter metadata before extraction.

R50a. `init --adapter codex --from-archive <path>` MUST NOT require a separate user-facing metadata path in this slice.

R50b. The CLI package MUST include bundled adapter metadata for the package's default compatible Codex adapter release.

R50c. For package version `0.1.3`, the default compatible adapter release MUST be `v0.1.3`.

R50d. If bundled adapter metadata for the requested adapter and release is unavailable, the command MUST return status `blocked`, exit code `2`, and blocker code `metadata-unavailable`.

R50e. The bundled release index MUST record the bundled adapter metadata filename and SHA-256 for the package's default compatible Codex adapter release.

R50f. The CLI MUST verify bundled adapter metadata bytes against the bundled release index SHA-256 before parsing metadata for either network or local archive installation.

R51. Release metadata MUST include:

```yaml
schema_version: 1
release:
  version: v0.1.3
  source_repository: xiongxianfei/rigorloop
  source_commit: "<sha>"
  release_tag: v0.1.3
  published_at: "YYYY-MM-DD"
metadata:
  url: "<official metadata URL>"
  sha256: "<sha256>"
artifacts:
  - adapter: codex
    archive: rigorloop-adapter-codex-v0.1.3.zip
    url: "<official archive URL>"
    sha256: "<sha256>"
    size_bytes: 12345
    install_root: ".agents/skills"
    tree_hash_algorithm: rigorloop-tree-hash-v1
    tree_sha256: "<sha256>"
validation:
  command: "python scripts/validate-adapters.py --root <release-output-dir> --version v0.1.3"
  result: pass
```

R52. The command MUST reject metadata whose `release.source_repository` is not `xiongxianfei/rigorloop`.

R53. The command MUST reject metadata whose artifact `adapter` is not `codex`.

R54. The command MUST reject metadata whose Codex artifact `install_root` is not `.agents/skills`.

R55. The command MUST reject metadata whose `validation.result` is not `pass`.

R56. The command MUST verify archive SHA-256 before extraction.

R57. The command MUST verify `size_bytes` when the metadata provides it.

R58. The command MUST compute the installed Codex adapter tree hash after extraction using `rigorloop-tree-hash-v1`.

R59. The command MUST compare the computed tree hash with metadata `tree_sha256` when metadata provides it.

R60. Checksum mismatch, size mismatch, tree-hash mismatch, invalid metadata hash, invalid metadata schema, or path traversal MUST produce status `error`.

R61. Missing bundled metadata, unknown adapter, incompatible release version, or unavailable official archive source MUST produce status `blocked` unless the failure is an unexpected internal error.

R61a. Metadata unavailable for a requested local archive adapter or release MUST produce status `blocked`, exit code `2`, and blocker code `metadata-unavailable`.

R61b. Adapter unknown MUST produce status `blocked`, exit code `2`, and blocker code `adapter-unknown`.

R61c. Archive SHA mismatch, archive size mismatch, installed tree-hash mismatch, metadata-hash mismatch, and archive path traversal MUST produce status `error` and exit code `3`.

### Planned lockfile output

R62. This slice MUST NOT create or modify `rigorloop.lock`.

R63. JSON output for `init --adapter codex` MUST include `planned_lockfile` whenever adapter archive metadata or adapter install planning is available.

R64. Human output for actual init SHOULD state that `rigorloop.lock` was not written because the durable lockfile contract is not approved.

R65. `planned_lockfile` MUST identify `tree_hash_algorithm: rigorloop-tree-hash-v1`.

R66. `planned_lockfile` MUST identify the Codex adapter source, archive name, expected archive SHA-256 when known, install root, and expected or computed tree SHA-256 when known.

R67. Successful actual init MUST emit a warning with code `lockfile-spec-not-approved` when `planned_lockfile` is available and `rigorloop.lock` is intentionally not written.

### Tree hash

R68. `rigorloop-tree-hash-v1` MUST hash regular files only.

R69. `rigorloop-tree-hash-v1` MUST exclude directories, symlinks, modification times, creation times, owner, group, absolute paths, the lockfile itself, and temporary files.

R70. Tree-hash paths MUST be relative to the generated output root, use POSIX `/`, have no leading `./`, have no trailing slash, be UTF-8 strings, and be sorted lexicographically.

R71. Text generated by RigorLoop MUST be normalized to UTF-8 with LF line endings, with a UTF-8 BOM removed if present, without trimming whitespace or semantically normalizing Markdown.

R72. Binary files MUST be hashed as raw bytes.

R73. Each file hash MUST be `sha256(normalized_file_bytes)`.

R74. The tree manifest MUST be:

```text
rigorloop-tree-hash-v1\n
<relative_path>\t<file_sha256>\n
<relative_path>\t<file_sha256>\n
...
```

R75. The tree hash MUST be `sha256(utf8(canonical_manifest))`.

### Publication boundary

R76. This spec MUST NOT authorize public npm publication.

R77. Public npm publication MUST remain blocked until a release-policy or hardening artifact accepts package contents, trusted publishing, provenance, release workflow controls, dependency policy, lifecycle-script policy, versioning, and rollback behavior.

R78. The npm package MUST NOT bundle adapter archives as authored source.

R79. The npm package MUST NOT treat generated adapter files as canonical skill source.

## Inputs and outputs

Inputs:

- command-line arguments for `rigorloop --help`, `rigorloop version`, and `rigorloop init --adapter codex`;
- current working directory or explicit target directory if implemented;
- package name and version;
- bundled adapter metadata shipped with the installed CLI package;
- official GitHub release archive URLs named by the bundled adapter metadata;
- optional local Codex adapter archive path from `--from-archive`;
- existing project files, including `rigorloop.yaml` and `.agents/skills/**` when present.

Outputs:

- human-readable stdout for help, version, and non-JSON init;
- JSON stdout for `--json`;
- `rigorloop.yaml` when actual init succeeds and the file is absent;
- `.agents/skills/**` generated Codex adapter files when actual adapter installation succeeds;
- no durable `rigorloop.lock` writes;
- exit codes defined by this spec.

## State and invariants

- `skills/` remains the only authored skill source.
- Generated Codex adapter files remain generated output, not canonical source.
- Adapter release archives remain GitHub release artifacts, not bundled authored npm source.
- `rigorloop.lock` remains absent or untouched until a lockfile spec is accepted.
- The CLI does not claim proposal acceptance, branch readiness, PR-body readiness, PR-open readiness, or validation success.
- All mutation behavior is planned before writing and is non-destructive by default.
- JSON output remains parseable and stable at the envelope level for this schema version.

## Error and boundary behavior

- Invalid command syntax exits `4`.
- Unsupported adapter values exit `2` with status `blocked`.
- Missing bundled metadata exits `2` with status `blocked`.
- Unknown adapter metadata exits `2` with status `blocked`.
- Release version mismatch exits `2` with status `blocked`.
- Checksum, size, tree-hash, metadata-hash, metadata schema, or archive traversal failures exit `3` with status `error`.
- Existing user-file overwrite conflicts exit `5`.
- Permission failures while writing files exit `1` unless they can be classified as overwrite refusal.
- Network failure while fetching the official adapter archive exits `2` with status `blocked` when retrying later or using `--from-archive` is a valid next action.
- Dry-run failures must not leave filesystem changes.
- Actual init failures must not silently ignore partial writes.

## Compatibility and migration

This spec introduces an additive CLI package candidate and does not remove existing repository scripts, adapter generation commands, release checks, or `dist/adapters/README.md` install guidance.

Existing users may continue installing adapters from documented release archives without using the CLI.

The first implementation may be tested locally or from a packed package artifact before public npm publication. Public npm publication requires a later accepted release-policy or hardening artifact.

Rollback before public publication is to remove or disable the package candidate and keep existing release-archive install guidance.

Rollback after a later publication is outside this spec and must be defined by the npm release hardening slice before publication occurs.

## Observability

- Human init output MUST summarize created, skipped, blocked, and warning items.
- JSON init output MUST include action and artifact lists sufficient to reconstruct planned or completed mutations.
- Debug output MUST avoid printing large adapter file bodies by default.
- Error output MUST name the failed verification step when metadata, archive, size, path, or tree-hash verification fails.
- Commands MUST NOT claim hosted CI, release validation, or workflow validation passed unless those checks actually ran.

## Security and privacy

- The command MUST NOT require secrets or credentials for normal public release-archive installation.
- The command MUST NOT print secrets from environment variables or config files.
- The command MUST NOT send project file contents over the network.
- Network access, when used, MUST be limited to fetching official adapter archives for the requested adapter.
- Archive extraction MUST defend against path traversal and symlink writes.
- The command SHOULD use minimal runtime dependencies; dependency policy is finalized by the release hardening slice before publication.

## Accessibility and UX

This slice has no graphical UI.

Human CLI output MUST be readable without color. Errors MUST include a concise reason and, when possible, a next action.

## Performance expectations

Dry-run planning for an empty target project SHOULD complete without downloading an adapter archive when bundled adapter metadata is enough to report planned actions.

Actual init SHOULD avoid reading or hashing files outside the target scaffold paths and selected archive extraction root.

The command MUST stream or bound archive processing so routine Codex adapter installation does not require loading the full archive and all extracted file bodies into user-visible output.

## Edge cases

1. Existing `rigorloop.yaml` is present and valid: init reports it as existing and does not overwrite it by default.
2. Existing `rigorloop.yaml` is present but incompatible: init blocks with an actionable config error rather than merging blindly.
3. `.agents/skills` exists but is empty: init may install into it after planning confirms no file conflicts.
4. `.agents/skills` contains unrelated files: init refuses to overwrite or delete them.
5. Release metadata lists multiple adapters: init selects only the `codex` artifact.
6. Bundled adapter metadata for the requested adapter and release is unavailable: init blocks with exit code `2` and blocker code `metadata-unavailable`.
7. `--from-archive` path does not exist: init exits `4` for invalid input.
8. Archive contains symlinks: init rejects the archive.
9. Archive contains files outside `.agents/skills`: init rejects the archive.
10. The package version cannot be determined: `version` and JSON package fields report an error rather than guessing.
11. `--json` and `--quiet` are combined: JSON output remains complete.
12. `NO_COLOR` is set with human output: no ANSI color is emitted.
13. `--from-archive` names a `v0.1.3` archive while the installed package expects another release and no compatibility rule exists: init blocks with exit code `2`.

## Non-goals

- No `rigorloop new-change` behavior.
- No `rigorloop status` behavior.
- No `rigorloop validate` behavior.
- No durable `rigorloop.lock` writes.
- No workflow YAML source-of-truth behavior.
- No generated workflow docs, diagrams, validator tables, or frozen drift checks.
- No npm trusted publishing or public publication workflow.
- No Claude Code or opencode adapter installation.
- No adapter archive generation.
- No TypeScript port of repository-owned Python validators.
- No readiness claims for implementation, review, verification, or PR stages.

## Acceptance criteria

- AC1. A local package execution can run `rigorloop --help` and show only the in-scope command surface.
- AC2. A local package execution can run `rigorloop version` and report `@xiongxianfei/rigorloop` plus the package version.
- AC3. `rigorloop init --adapter codex --dry-run --json` emits the stable JSON envelope, planned writes, and `planned_lockfile` without writing files.
- AC4. Actual `rigorloop init --adapter codex` creates `rigorloop.yaml` and installs verified Codex adapter files from an approved release archive when all metadata, checksum, size, path, and tree-hash checks pass.
- AC5. Actual init does not create or modify `rigorloop.lock`.
- AC6. Existing user-file conflicts are refused with exit code `5`.
- AC7. Missing metadata, unknown adapter metadata, and release-version mismatch produce status `blocked`.
- AC8. Checksum mismatch, size mismatch, tree-hash mismatch, and archive path traversal produce status `error`.
- AC9. Unsupported adapters are rejected and no non-Codex adapter files are written.
- AC10. Public npm publication remains blocked by documentation or release checks until the separate release hardening slice is accepted.
- AC11. `--from-archive` succeeds with matching bundled adapter metadata and fails with `metadata-unavailable` when bundled metadata is absent for the requested adapter and release.
