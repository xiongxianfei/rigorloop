# RigorLoop Release Process Contract

## Status

accepted

## Problem

RigorLoop has improved its lifecycle discipline for proposing, specifying, implementing, reviewing, verifying, and merging changes. It has also had release-pipeline hardening work in the past. What is still missing is a standing release-process contract: a durable definition of how an already-merged RigorLoop version is packaged, published, verified, and recorded.

A routine publish is not the same kind of work as a normal RigorLoop change. The lifecycle handles decisions about what to build, while a publish is an operation on already-decided, already-reviewed, already-merged work. The process for publishing is contract-worthy once; each routine publish should then execute that standing process without a new proposal or plan.

Without a standing contract, each release risks being reconstructed from memory, prior PRs, or ad hoc notes. That creates several failure modes:

| Gap | Risk |
| --- | --- |
| No standing release gate | A release may publish without a consistent full validation boundary. |
| No version-decision rule | Patch, minor, major, prerelease, and republish decisions may be improvised. |
| No packaged-output freshness rule | Release artifacts may drift from canonical source. |
| No registry-verification rule | Publication may be assumed rather than verified from the registry. |
| No evidence artifact shape | Release proof may be incomplete, scattered, or hard to review later. |
| No routine-publish boundary | Contributors may over-apply proposal/spec/plan ceremony to every publish. |

The goal is not to make releases heavy. The goal is to define the process once so each routine publish becomes a deterministic checklist execution with durable evidence.

## Goals

- Define a standing RigorLoop release process.
- Distinguish the release process contract from each individual publish operation.
- Avoid proposal/spec/plan ceremony for routine publishes that create no new product or implementation decision.
- Require a full release gate before publish.
- Require publish from canonical, regenerated, validated source.
- Define version-decision rules for patch, minor, major, prerelease, and no-op publish decisions.
- Define npm publication requirements, including authentication/provenance policy.
- Define post-publish verification against the public registry.
- Define a durable release evidence artifact.
- Define rollback and recovery behavior for failed publish, bad package content, registry mismatch, and post-publish smoke failure.
- Preserve existing lifecycle semantics: implementation changes still go through proposal/spec/plan/etc. before merge; release runs only after the release candidate is already merged or otherwise explicitly approved.

## Non-goals

- Do not require a new proposal, spec, or plan for every routine publish.
- Do not use release evidence to bypass code review, verify, or PR readiness.
- Do not publish unmerged or unreviewed code through this process.
- Do not change package contents, public skill behavior, adapter layout, CLI behavior, lockfile semantics, or release archive trust boundaries in this proposal.
- Do not define a deploy process for non-npm infrastructure.
- Do not define a backport policy beyond release-version handling.
- Do not make external CI logs the only durable release evidence.
- Do not rely on local machine state, untracked build output, or hand-edited generated artifacts.
- Do not store npm tokens, credentials, OTPs, hostnames, or private environment dumps in release evidence.
- Do not claim the first release-process implementation is itself a release unless the release evidence is generated by executing the approved process.

## Vision fit

fits the current vision

RigorLoop's artifact-first workflow depends on durable proof that important transitions happened correctly. Publishing a package is one of those transitions. The release process should make the common path deterministic while preserving enough evidence to reconstruct what was published, from what source, by what gate, and with what verification.

This proposal is falsified if any of these occur:

```text
- a routine publish requires full proposal/spec/plan ceremony without a new decision;
- a release publishes source that does not match canonical repository state;
- generated skills/adapters are stale at publish time;
- registry publication is not verified after publish;
- package installation/smoke proof is missing;
- release evidence cannot reconstruct version, commit, package, registry result,
  smoke result, and artifact checks;
- credentials, tokens, OTPs, or private environment values are recorded.
```

## Context

The process distinction is:

| Thing | Treatment |
| --- | --- |
| Release process | Worth defining once through proposal/spec because it creates a durable operating contract. |
| Individual routine publish | Operation that executes the process; no new proposal/plan unless the release itself introduces a new decision. |

The release evidence record is the publish's durable artifact, analogous to a verify report: publication record, smoke proof, install proof, and registry verification.

npm's current trusted publishing documentation says trusted publishing uses OIDC from supported CI/CD workflows and eliminates long-lived npm tokens. npm also recommends trusted publishing over tokens for CI/CD package publishing. [npm trusted publishing][1]

npm supports package provenance. Its docs state that trusted publishing automatically generates provenance attestations, while `npm publish --provenance` is the CLI-side path for supported cloud CI/CD publishing outside trusted publishing. [npm provenance][2]

npm's current package security model also requires either 2FA or a granular access token with bypass 2FA for package creation and publication, and npm documents a stricter option that requires 2FA and disallows tokens. [npm 2FA][3]

| Automated release CLI | separate proposal | Process should exist before scaffolding automation. |
| Backport/LTS policy | separate proposal | Version support policy is broader than publish mechanics. |

## Options Considered

### Option 1: Do nothing

Keep releases as ad hoc maintainer operations.

Pros:

- No process work.
- Fast for maintainers who already know the steps.

Cons:

- Release quality depends on memory.
- Evidence may be inconsistent.
- Registry verification may be skipped.
- Future maintainers lack a durable contract.

### Option 2: Require full proposal/spec/plan for every release

Treat every publish as a normal lifecycle change.

Pros:

- Maximum ceremony and review.
- Every publish has a lifecycle trail.

Cons:

- Wrong abstraction for routine publishes.
- A routine publish makes no new product or implementation decision.
- High ceremony encourages bypasses.
- Duplicates the same checklist for every version.

Rejected for routine publishes.

### Option 3: Define the release process once, then execute it per release

Create a standing release contract and evidence artifact. Each routine publish executes the contract without a new proposal or plan.

Pros:

- Durable process.
- Low ceremony per publish.
- Consistent evidence.
- Clear boundary between changed code and publish operation.
- Matches RigorLoop's artifact discipline.

Cons:

- Requires initial spec/test-spec/plan work.
- Requires deciding CI/trusted publishing details.

Recommended.

### Option 4: Fully automate release immediately

Add release CLI commands, GitHub Actions, changelog generation, npm publish, tag creation, and evidence writing in one change.

Pros:

- Strong end-state.
- Reduces manual error after implementation.

Cons:

- Too much first-slice surface.
- Release automation can publish real packages if misconfigured.
- Harder to review safely.
- Process contract should come before automation.

Defer broad automation until the contract lands.

## Recommended Direction

Choose Option 3.

Create a standing release-process contract. After it is approved, a routine publish is not a new proposal or plan; it is an operation that executes the approved release checklist and records a release evidence artifact.

Use this rule:

```text
Contract the release process once.
Execute routine publishes many times.
```

## Release-process contract

### Release types

| Release type | Meaning | Requires new proposal/plan? |
| --- | --- | --- |
| Routine patch/minor publish | Publishes already-merged reviewed changes under the standing process. | no |
| Prerelease publish | Publishes release candidate, beta, alpha, or next tag under standing process. | no, unless policy changes |
| First release using new process | Establishes or validates release contract and evidence shape. | yes, for process creation |
| Breaking release | Version may be major, but release operation itself is still routine if breaking change was already reviewed upstream. | no for publish; upstream change needed lifecycle |
| New package name / new npm scope / new adapter target | Changes release surface. | yes |
| Release process change | Changes gate, evidence, auth, provenance, versioning, or publish mechanics. | yes |
| Emergency hotfix publish | May execute routine process with emergency evidence notes; upstream hotfix change still needs required review or approved emergency path. | maybe, depending on code-change path |
| Republish attempt after failed publish | Operational recovery; record in evidence. | no if package version was not published; yes if policy changes |

### Version decision rules

Before publishing, determine release version and tag.

| Decision | Rule |
| --- | --- |
| No source/package change | Do not publish. |
| Patch release | Backward-compatible fixes, docs, validation, packaging, or internal workflow changes that do not add public API/behavior. |
| Minor release | Backward-compatible public behavior, CLI, skill, adapter, or packaging capability additions. |
| Major release | Breaking public API, skill contract, adapter contract, install root, archive format, or CLI behavior change. |
| Prerelease | Use when publish is intended for validation before stable adoption. |
| npm dist-tag | Use `latest` only for stable release; use `next`, `beta`, `alpha`, or equivalent for prerelease. |

The version decision should be recorded in the release evidence.

### Release gate

A release candidate should pass the full release gate before publish.

Minimum gate:

```text
1. clean working tree except intentional release artifacts;
2. release version selected and recorded;
3. changelog/release notes prepared or explicitly not required;
4. package contents inspected;
5. generated skills/adapters rebuilt or validated from canonical source;
6. test suite / selected CI / broad smoke pass;
7. package build succeeds;
8. npm pack or equivalent package preview inspected;
9. local install smoke passes from packed artifact;
10. no unresolved release blockers;
11. publish authentication/provenance path selected;
12. release evidence path prepared.
```

The release gate may be automated, but automation should not hide the gate's evidence.

### Generated artifact currency

Before publish:

```text
generated output should be current from canonical source
```

For RigorLoop, this includes at minimum:

```text
skills/
dist/adapters/ or generated adapter archives
package files
lockfiles or manifest files relevant to packaging
```

Rules:

```text
- Do not hand-edit generated adapter output.
- Do not publish from stale generated artifacts.
- If tracked generated trees are intentionally stale or not authoritative,
  record that boundary and validate temporary generated release output instead.
- The release evidence should say which generated-output proof was used.
```

### npm publication policy

Preferred publication path:

```text
trusted publishing from CI/CD using OIDC
```

Reason: npm's current guidance says trusted publishing eliminates long-lived npm tokens and recommends trusted publishing over tokens for CI/CD package publishing. [npm trusted publishing][1]

Publication policy:

| Topic | Rule |
| --- | --- |
| Authentication | Prefer trusted publishing/OIDC. Use granular token fallback only when trusted publishing is not available. |
| 2FA | Preserve npm's 2FA requirements; do not record OTPs or secrets. npm currently requires 2FA or granular access tokens with bypass 2FA for creating and publishing packages. [npm 2FA][3] |
| Provenance | Prefer automatic provenance through trusted publishing; otherwise use supported provenance generation when publishing from supported CI/CD. [npm provenance][2] |
| Package contents | Run package preview and inspect included files before publish. |
| Dist-tag | Publish stable releases to `latest`; prereleases to a non-latest tag unless explicitly approved. |
| Manual publish | Allowed only as fallback; record reason, local gate evidence, and post-publish registry proof. |
| Staged publishing | Optional follow-up for higher-risk workflows. npm documents staged publishing with trusted publishers as a way to submit a package for review before it goes live. [npm staged publishing][4] |

## Routine publish operation

A routine publish follows this checklist.

### 1. Preflight

Record:

```text
release version
release type
source commit
branch
package name
npm dist-tag
publish path: trusted-publishing | manual-token | manual-2fa | staged
```

Stop if:

```text
working tree is dirty in unapproved ways
version already exists on npm
release notes are missing when required
generated output is stale
release blockers remain open
```

### 2. Build and package preview

Run package build and preview:

```bash
npm pack --dry-run
npm pack
```

Record:

```text
package filename
package size/integrity if available
included-file review
unexpected included/excluded files
```

### 3. Local install smoke

Install from the packed artifact into a temporary clean project.

Minimum smoke:

```text
package installs
CLI entrypoint responds if package has one
published skills/adapters are present if package owns them
adapter install proof runs if adapters are part of package
no repository-local source dependency is required
```

### 4. Publish

Preferred:

```text
CI trusted publishing
```

Fallback:

```text
npm publish
```

or:

```text
npm publish --provenance
```

when using a supported CI/CD provenance path outside trusted publishing. npm documents `--provenance` for supported cloud CI/CD systems. [npm provenance][2]

Record:

```text
publish command family, not secrets
registry result
dist-tag
version
time
provenance status
```

### 5. Post-publish registry verification

After publish, verify from the registry, not from local build output.

Minimum verification:

```bash
npm view <package>@<version> version
npm view <package>@<version> dist-tags
npm view <package>@<version> dist.integrity
```

Then run fresh install smoke:

```bash
npm install <package>@<version>
# or
npx <package>@<version> <smoke command>
```

Use exact package commands in the spec/plan after package name and CLI surface are confirmed.

### 6. Release evidence closeout

Write the release evidence artifact and update any release index or changelog required by the release spec.

## Exception handling

### Failed before publish

Examples:

```text
package preview wrong
local install smoke fails
tests fail
generated output stale
```

Required action:

```text
stop
record failed-before-publish evidence
fix through normal lifecycle if source changes are required
do not publish
```

### Failed during publish

Examples:

```text
npm auth failure
2FA failure
network failure
version already exists
provenance failure
```

Required action:

```text
record failure
determine whether version was published
if not published, rerun after fixing operational issue
if published partially or uncertain, verify registry before retry
```

### Failed after publish

Examples:

```text
fresh install smoke fails
wrong dist-tag
package contents wrong
```

Required action:

```text
record failed-after-publish
assess severity
fix-forward with new version when package content is wrong
correct dist-tag if only tag is wrong and policy allows
deprecate bad version if necessary
do not attempt to overwrite an already-published npm version
```

### Emergency release

An emergency release may use the routine publish process, but evidence should record:

```text
why it is emergency
which upstream review/approval path was used
which normal gates were preserved
which gates were deferred, if any
owner-approved follow-up for deferred gates
```

## Expected Behavior Changes

- Routine publishes no longer require ad hoc process reconstruction.
- Each release has a durable evidence artifact.
- Release gates become explicit and repeatable.
- Publishing from stale generated output becomes a detectable release-process failure.
- npm authentication/provenance policy is explicit.
- Routine publishes remain lightweight operations rather than full lifecycle changes.

## Rollout and Rollback

Rollout:

1. Approve this proposal.
2. Write release-process spec.
3. Write test spec / checklist validation for release evidence.
4. Plan implementation with separate reviewable slices for the evidence template, gate commands, npm publish path, evidence validation, and dry-run rehearsal.
5. Code-review each milestone.
6. Rehearse release process without publishing.
7. Use the process for the next real release.
8. Record first release evidence.

Rollback:

- If process spec is wrong, revise the release process through normal lifecycle.
- If CI trusted publishing configuration fails, use documented manual fallback only after full local gate and evidence.
- If a release evidence artifact is malformed, fix evidence through reviewable artifact correction; do not change the published package unless the package itself is wrong.
- If a bad package is published, use npm dist-tag correction, deprecation, or fix-forward release as appropriate; do not try to overwrite an existing published version.

## Risks and Mitigations

| Risk | Mitigation |
| --- | --- |
| Routine release becomes over-ceremonial | Explicitly state routine publishes execute the process without proposal/plan. |
| Release evidence leaks secrets | Ban tokens, OTPs, credentials, hostnames, and environment dumps. |
| Stale generated output is published | Gate requires generated output rebuild/validation from canonical source. |
| npm auth path is brittle | Prefer trusted publishing/OIDC; keep documented manual fallback. |
| Provenance is missing | Record provenance mode and reason; prefer trusted publishing automatic provenance. |
| Wrong dist-tag | Preflight and post-publish registry verification include dist-tag checks. |
| Bad package contents | npm pack preview and packed install smoke before publish. |
| Published package fails fresh install | Post-publish smoke and recovery policy. |
| Process changes without review | Release-process changes require normal lifecycle treatment. |
